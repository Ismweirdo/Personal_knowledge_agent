# PostgreSQL、SQLAlchemy、Alembic 与 pgvector 全解

## 1. 为什么统一使用 PostgreSQL + pgvector

项目需要保存用户、知识库、知识源、任务、会话、Chunk 和向量。PostgreSQL 事务和 SQL 能力成熟，pgvector 直接支持向量距离与元数据过滤，使 MVP 不需要同时运维业务数据库和独立向量库。

好处是部署简单、业务过滤和向量检索可以在一个查询中完成，事务与备份体系统一。代价是超大规模向量吞吐、分片和专用检索能力不如 Milvus/Qdrant 等独立向量数据库。个人知识库优先简单可靠，规模明显增长后再迁移。

## 2. 事务与隔离级别

事务满足 ACID。PostgreSQL 使用 MVCC，让读操作通过数据版本减少读写阻塞。默认 `READ COMMITTED` 每条语句读取开始时已提交的数据；同一事务两次查询可能看到不同结果。`REPEATABLE READ` 提供事务级快照，但并发冲突可能需要重试。

项目事务边界：创建知识源和同步任务可以同事务提交；外部抓取、Embedding 和模型调用不能放进长事务。新版本 Chunk 应分批写入，全部成功后用短事务切换 `active_version`，失败保留旧版本。

常见问题：事务过长导致连接占用、旧版本无法清理；捕获异常却提交部分数据；在异步任务中共享 Session。解决方式是短事务、异常回滚、每个请求/任务独立 Session、状态机和补偿清理。

## 3. SQLAlchemy 2 的实际用法

SQLAlchemy 提供连接池、SQL 构造、ORM 映射和 Unit of Work。项目可以使用异步 `AsyncSession`，但 Session 不是线程/协程安全对象，必须按请求或任务创建。

推荐分层：Repository 负责查询与持久化，Application Service 负责事务和业务规则，API 不直接拼复杂 SQL。简单 CRUD 不需要为了“分层”创建大量空接口，关键查询应能看到实际 SQL 和执行计划。

常见问题：

- N+1 查询：使用显式 join/selectinload，检查 SQL 日志。
- 提交后访问延迟加载字段触发异步错误：在 Session 生命周期内加载所需字段，响应 DTO 显式转换。
- 连接池耗尽：检查未关闭 Session、长事务和并发上限；设置 pool size、timeout 和指标。
- 批量插入逐条 flush：使用批量 insert/upsert，并控制批次大小。

## 4. Alembic 为什么必须使用

Alembic 记录数据库 Schema 版本，支持升级、回滚和多人协作。生产部署不能依赖 ORM 启动时自动建表，否则变更不可审计且可能破坏数据。

迁移脚本需要人工审查，尤其是非空列、索引、大表字段类型和数据回填。大表建索引可能锁表，应评估 PostgreSQL `CREATE INDEX CONCURRENTLY`。先兼容旧代码的扩展式迁移，再切换读取，最后删除旧字段，比一次破坏性修改安全。

## 5. pgvector 原理

Embedding 将文本映射为高维向量，语义相近文本在向量空间距离较近。常用距离：余弦距离关注方向，内积适合已归一化向量，L2 关注欧氏距离。查询和索引必须使用与模型/数据一致的距离度量。

精确检索会扫描全部候选向量，结果准确但规模大时慢。HNSW 构建多层近邻图，查询快、召回高、索引占用较大且构建较慢；IVFFlat 先聚类再搜索部分列表，构建较轻但需要训练和调参，数据量太小时效果不稳定。

项目查询必须在数据库中先限制 `user_id`、`kb_id`、`source_version` 和 READY 状态，不能全库取 Top-K 后在 Python 过滤，否则存在数据泄露且正确候选可能被挤出 Top-K。

## 6. 索引与 SQL 优化

业务索引围绕查询设计：知识源唯一约束可使用 `(user_id, kb_id, normalized_url)`；任务扫描使用 `(status, next_retry_at)`；消息历史使用 `(conversation_id, created_at)`；Chunk 元数据过滤字段需配普通索引。

使用 `EXPLAIN (ANALYZE, BUFFERS)` 观察实际行数、扫描方式、排序和缓存命中。不要仅凭“建了索引”判断优化成功。低选择性字段、对列使用函数、隐式类型转换和前导模糊匹配可能无法有效使用 B-tree。

## 7. 常见故障

- Embedding 模型更换后维度不一致：知识库记录 embedding model/version，新模型单独重建，禁止混存。
- 删除知识源但向量残留：使用版本/外键和后台清理任务，检索强制 READY/active 过滤。
- HNSW 召回下降：对比精确检索基线，调整搜索参数并用固定评测集验证。
- 元数据 JSON 滥用：高频过滤字段独立成列，JSONB 保存低频扩展属性。
- 数据库既做业务又做向量导致资源竞争：监控连接、CPU、I/O；隔离连接池、限制并发，规模增长后拆向量库。

## 8. 面试追问

**为什么不直接用 Milvus？** 当前规模下 pgvector 简化事务、过滤、备份和部署；当向量规模、吞吐或独立扩缩容成为瓶颈再拆分。

**ORM 会不会慢？** ORM 的问题通常是错误查询方式而不是抽象本身。关键是查看生成 SQL、避免 N+1、批量操作并用执行计划验证。

**向量索引为什么是近似检索？** 为避免全量距离计算，以少量准确率换取显著速度，需要通过 Recall@K 对照精确检索评估。
