# Redis、认证安全、测试与部署全解

## 1. Redis 在项目中的作用

Redis 用于限流、短期会话/热点数据缓存、分布式任务协调和临时状态，不作为不可恢复业务数据的唯一存储。JWT 校验本身不必每次查 Redis；需要注销/黑名单时才保存 token ID 或用户版本。

Redis 单线程执行命令（网络 I/O 已演进为多线程）使单条命令原子，但多条命令组合并不自动原子。Lua 可将检查与更新一次执行，适合滑动窗口限流和锁释放。

核心原理：Redis 将常用数据结构保存在内存，通过事件循环处理大量连接；命令执行路径短，避免传统磁盘数据库的页读取和复杂查询开销。持久化、复制和集群提高可恢复性与容量，但不能把缓存直接当成绝对可靠的业务数据库。

## 2. 缓存模式与常见问题

Cache-Aside：读缓存，未命中读数据库并回填；更新数据库后删除缓存。项目可缓存知识库列表、知识源状态和短期会话摘要，向量真相仍在 PostgreSQL。

- 缓存穿透：查询不存在数据，使用参数校验、空值短缓存或布隆过滤器。
- 缓存击穿：热点 Key 过期，大量请求回源；使用互斥重建、逻辑过期或提前刷新。
- 缓存雪崩：大量 Key 同时过期；TTL 加随机值、限流降级、多级缓存。
- 大 Key/热 Key：拆分数据、分页、限制 value，使用监控识别。
- Redis 不可用：有限回源数据库，限流保护，禁止无限重试拖垮服务。

## 3. 限流

登录按 IP+账号，上传/同步按用户并发数，问答按用户分钟次数和同时流数限制。固定窗口简单但边界可能突发；滑动窗口更准确但存储更多；令牌桶允许可控突发，适合 API。

分布式限流使用 Redis Lua 原子执行。限流失败时根据接口价值选择 fail-open 或 fail-closed：登录安全通常偏关闭，普通读取可有限放行，模型付费接口应保护成本。

## 4. JWT 与权限

JWT 由 Header、Payload、Signature 构成，Payload 只编码不加密，不能存密码和敏感资料。签名用于防篡改；使用强随机密钥、明确算法、过期时间、issuer/audience，并只通过 HTTPS。

认证只证明“你是谁”，授权还要判断“是否拥有这个知识库/会话”。所有 Service/Repository 查询强制带当前 `user_id`，不能先按 ID 查询再相信前端。向量检索同样必须带用户和知识库过滤。

Access Token 短期有效，Refresh Token 可轮换并保存哈希。登出、封禁和改密可通过用户 token version 或黑名单实现。不要在日志中打印完整 Token。

## 5. Web 与模型安全

- SSRF：URL 抓取禁止内网、loopback、云元数据，重定向后重新验证。
- 文件安全：MIME/大小校验、UUID 文件名、路径隔离、解析资源限制。
- XSS：Markdown/模型输出按白名单清理，前端不直接执行 HTML。
- Prompt Injection：资料是不可信数据，工具权限服务端校验。
- 数据泄露：日志不记录完整文档、Prompt 和密钥；错误响应不返回堆栈。
- 成本攻击：上传、同步、Embedding 和问答分别配额/限流，设置 Token 上限和超时。

## 6. pytest 测试体系

单元测试覆盖 Chunk、哈希规范化、Prompt、权限、状态机和错误分类；使用 Fake 模型避免测试依赖真实付费 API。异步 API 使用 pytest-asyncio 与 httpx。

集成测试用 Testcontainers 启动 PostgreSQL/pgvector 和 Redis，覆盖事务、元数据过滤、版本切换和缓存降级。端到端测试从添加知识源到流式回答。故障测试覆盖 429、超时、模型中断、重复调度、数据库回滚和 Redis 不可用。

测试金字塔中单元测试最多，集成测试验证边界，少量 E2E 验证主流程。不要只追求覆盖率；权限隔离、数据一致性和失败恢复比简单 getter 更重要。

## 7. Docker Compose 与 Nginx

Compose 运行 `frontend/backend-python/postgres/redis/nginx`。数据库和上传目录挂载持久卷；服务设置健康检查、资源限制和重启策略；密钥来自服务器环境变量，不写进镜像。

Nginx 负责 HTTPS、静态资源和反向代理。SSE 路径关闭缓冲、增加读取超时，并传递必要头部。只开放 80/443，数据库与 Redis 留在内部网络。

镜像采用多阶段构建、固定依赖版本、非 root 用户和小型基础镜像。数据库迁移在可控部署步骤执行，不让多个实例同时无序迁移。

## 8. 可观测性

日志使用 requestId、taskId、conversationId 和 sourceId 串联链路。指标包括 API P95/P99、同步成功率、任务积压、检索耗时、Recall@K 离线结果、首 Token 延迟、模型错误率、Token 用量、连接池和 Redis 命中率。

告警应面向用户影响：失败率、最老任务等待时间、模型 429、数据库连接池耗尽、磁盘空间和证书到期。只收集指标不设置阈值与处理手册不算可观测。

## 9. 常见问题与解决

- Redis/数据库连接池耗尽：检查未释放连接、长事务、并发上限和超时，增加池指标而非只扩大池。
- JWT 校验频繁失败：检查时间同步、issuer/audience、密钥轮换和 Header 格式，错误响应不暴露签名细节。
- SSE 上线后不流式：关闭代理缓冲，检查压缩、响应头和心跳。
- 容器可启动但依赖未就绪：readiness 检查数据库迁移和必要依赖，避免只依赖启动顺序。
- 测试偶发失败：移除真实模型和时间依赖，使用 Fake、固定时钟和隔离数据库。

## 10. 高频追问

**Redis 命令原子为什么还需要分布式锁？** 单命令原子不代表跨命令业务流程原子；但锁也不能替代数据库唯一约束和幂等。

**JWT 如何立即失效？** 短过期配合 Refresh Token，或使用黑名单/token version；代价是引入服务端状态。

**单元测试为什么不用真实模型？** 真实模型非确定、慢且收费。用 Fake 验证逻辑，用少量离线评测验证真实模型效果。

**Docker 容器重启数据为何不丢？** 数据必须存持久卷/外部数据库；容器文件系统是可替换的，且备份恢复需要单独验证。
