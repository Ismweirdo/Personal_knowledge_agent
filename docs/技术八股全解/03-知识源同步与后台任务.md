# 知识源同步、幂等与后台任务全解

## 1. 为什么知识源同步是项目核心

普通 RAG Demo 只支持一次上传文件。项目 1 面向长期使用的个人知识库，网页或知识文档变化后答案也必须更新，因此需要来源管理、变更检测、版本、失败恢复和定时同步。

收益是知识持续新鲜、减少重复 Embedding 和费用；代价是要处理网页结构变化、抓取权限、重复任务、半更新和删除同步。

## 2. Connector 抽象

统一连接器接口可以包含 `validate`、`fetch_metadata`、`fetch_content`、`normalize` 和 `source_identity`。文件、普通网页、未来 Notion/语雀使用不同实现，但向下游都输出统一文档对象：来源 ID、标题、正文、更新时间、内容类型和元数据。

好处是新增来源不修改入库主流程，测试可使用 Fake Connector。问题是过度抽象会隐藏来源差异，因此认证、分页、限流和删除语义仍由具体连接器负责。

## 3. HTTP 变更检测原理

`ETag` 是服务器为资源版本生成的标识；客户端下次带 `If-None-Match`，未变化可返回 304。`Last-Modified` 配合 `If-Modified-Since` 使用，但时间精度和服务端实现可能不可靠。

项目优先使用 ETag/Last-Modified；服务端不提供或行为异常时，抓取正文、去除导航/空白等噪声后计算 SHA-256。只比较原始 HTML 容易因广告、时间戳或无关 DOM 变化反复重建。

常见问题：弱 ETag、CDN 每次变化、动态页面无正文、反爬 403、重定向循环。解决方式是保存抓取诊断信息、限制跳转和域名、合理 User-Agent/超时、动态页面按需使用 Playwright，并遵守 robots/站点授权。

## 4. 增量更新与版本切换

安全流程：创建同步任务 -> 抓取并计算新版本 -> 内容未变直接成功 -> 解析和切分 -> 生成新版本 Chunk/向量 -> 校验数量 -> 短事务切换 active version -> 异步清理旧版本。

不能先删除旧向量再创建新向量，否则中途失败会让知识源不可用。版本切换类似蓝绿发布：读请求始终只查 active 且 READY 的版本。

更细粒度增量可为每个规范化段落计算哈希，复用未变化 Chunk 的向量。收益是降低费用；代价是段落移动、切分边界变化和元数据更新会增加匹配复杂度，MVP 先做来源级版本更稳妥。

## 5. 幂等与任务状态机

幂等表示同一操作执行一次或多次，最终业务效果相同。同步任务使用 `source_id + target_version/content_hash` 唯一约束；任务状态可为 `PENDING/RUNNING/SUCCEEDED/FAILED/CANCELLED`，通过条件更新抢占，避免两个 Worker 同时处理。

重试必须区分错误：网络超时、429、部分 5xx 可指数退避重试；无权限、文件损坏、格式不支持属于永久错误。重试前清理未完成版本或使用确定性 Chunk ID upsert。

只用进程内锁无法支持多实例。可以使用数据库唯一约束和状态抢占作为最终防线；Redis 锁只用于降低重复工作，不能替代业务幂等。

## 6. APScheduler 与任务队列

MVP 使用 APScheduler 触发“创建/领取同步任务”，实际状态保存在数据库。不要只把任务存在调度器内存，否则进程重启会丢失。

APScheduler 适合单机轻量定时任务。任务量、执行时间和独立扩缩容需求增加后，可迁移 Celery/RQ 或消息队列。迁移标准是持久队列、消费并发、重试/路由和 Worker 独立部署的收益超过运维成本。

## 7. 抓取与解析安全

URL 抓取必须防 SSRF：只允许 HTTP/HTTPS，解析 DNS 后禁止访问 loopback、内网、云元数据地址，限制重定向并在每次跳转后重新校验。设置响应大小、类型、连接/读取超时，禁止无限下载。

文件上传不能信任扩展名；校验 MIME、大小和解析器，使用服务端 UUID 文件名，禁止路径穿越。PDF/Office 解析应放在受限资源环境，处理压缩炸弹和超大页数。

项目中普通网页使用 `httpx` 获取响应，BeautifulSoup/readability 提取正文；动态渲染页面仅在必要时使用 Playwright，因为浏览器实例占用 CPU/内存且安全面更大。PDF 使用 PyMuPDF 按页提取文本并保留页码元数据，Markdown/TXT 优先直接解析。扫描版 PDF 没有文本层，需要 OCR，但 OCR 成本和误差较高，应作为后续能力而非默认流程。

解析器问题：PDF 字体编码可能导致乱码，表格阅读顺序可能错乱，Markdown 代码块不能按普通句号切分。解决方式是保存原始文件、按格式选择解析器、记录页/标题元数据、建立包含表格/代码/乱码样本的回归测试。

## 8. 故障排查

- 一直重复同步：检查正文规范化是否包含时间戳、随机 ID 或导航内容。
- 同步成功但问不到：检查 active version、Chunk 数量、Embedding 模型、元数据过滤和事务提交。
- 任务永久 RUNNING：增加 lease/heartbeat，超时任务由补偿器恢复为 PENDING/FAILED。
- 定时任务多实例重复：数据库抢占、唯一约束，必要时增加分布式锁。
- 页面删除后旧知识仍可查：同步删除/410 状态，停用来源版本并清理向量；谨慎区分临时 404。

## 9. 面试高频问答

**为什么内容哈希前要规范化？** 去掉无关变化，避免导航、空格和动态时间导致重复 Embedding；但规范化过度可能误删正文，需要保留原文和测试样本。

**如何保证更新期间可用？** 新版本完整构建后原子切换，读请求只查 active READY 版本，失败继续使用旧版本。

**分布式锁能否保证不重复？** 不能完全保证，锁可能过期或网络分区；唯一约束、状态机和幂等写入才是最终正确性保障。
