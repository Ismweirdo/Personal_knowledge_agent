# 开发文档：问题、决策与解决记录

本文记录 Personal Learning Knowledge Agent 从工程骨架到真实简历入库闭环过程中遇到的问题。每条记录包含现象、根因、方案、选择理由、验证和剩余风险，用于后续开发、排障、复盘和面试说明。

## 1. 项目边界从多租户知识库纠正为“我的个人 Agent”

**现象**：早期设计允许每个注册用户创建知识库、上传文档并构建自己的 Agent，这与产品目标不符。

**根因**：把“个人知识库产品”误解成“为每个用户提供个人知识库的 SaaS”。实际产品只有一个知识主体：管理员本人；访客只能与管理员的 Agent 对话。

**方案**：引入 `ADMIN/USER` 角色和知识库发布状态。知识库 CRUD、上传、同步、索引、图谱抽取、审核和学习管理统一依赖管理员授权；普通用户只拥有自己的会话。公开注册永远创建 USER，唯一管理员通过服务器交互式 CLI 创建或提升。

**选择理由**：授权规则在后端统一执行，不能依赖前端隐藏按钮。CLI 初始化避免攻击者抢先注册管理员邮箱。

**验证**：普通用户访问知识写接口返回 403；访客只能发现已发布 Agent；RAG 使用管理员作为知识 owner，消息按访客隔离。

## 2. 密钥不能进入 Git

**现象**：开发需要 DeepSeek Key 和 GitHub Models Token，但仓库是公开远程仓库。

**根因**：模型 SDK 需要真实凭据；把凭据写入配置文件会被提交历史永久保留。

**方案**：真实值只写入被 `.gitignore` 排除的 `.env`；仓库只提交 `.env.example` 占位符。每次提交前扫描暂存内容中的 `sk-` 和 `github_pat_` 模式。

**选择理由**：环境变量是当前单服务器部署最简单的秘密注入方式，后续可迁移到云 Secret Manager。

**验证**：`.env` 命中 Git ignore；PR 暂存扫描未发现真实 Token。

**剩余风险**：开发 Token 曾在对话中明文出现，正式部署前必须撤销并轮换。GitHub Token 应使用最小权限、短有效期的 fine-grained token。

## 3. DeepSeek 不承担 Embedding

**现象**：DeepSeek 对话接口可用，但不能假设同一端点提供 Embedding。

**根因**：聊天模型和向量模型是不同能力，提供方、模型名、限额和维度可能不同。

**方案**：拆分 `LLM_*` 与 `EMBEDDING_*` 配置和客户端。聊天使用 DeepSeek；Embedding 使用 GitHub Models 的 OpenAI 兼容端点和 `openai/text-embedding-3-small`。

**选择理由**：避免把提供方能力硬编码进业务，便于以后替换为 BGE-M3 或本地模型。

**验证**：GitHub Models 实际返回 1 个 1536 维向量；真实 pgvector 检索返回非空结果。

## 4. Embedding 维度必须与数据库一致

**现象**：不同 Embedding 模型可能返回 1024、1536 或其他维度，pgvector 列不能混存不同维度。

**方案**：当前锁定 `Vector(1536)`、`EMBEDDING_DIMENSIONS=1536` 和 HNSW cosine 索引。模型变更必须新增迁移、重建索引并重新向量化全部 Chunk。

**选择理由**：固定维度可以获得明确的数据库约束和索引行为；静默截断或填充会破坏相似度语义。

**验证**：真实 GitHub Models 调用返回 1536 维；PostgreSQL 集成测试验证真实 cosine 排序。

## 5. 知识更新不能先删除旧版本

**现象**：如果先删除旧 Chunk/向量，再生成新向量，模型超时会让线上知识突然不可用。

**方案**：来源版本不可变。新版本先进入 PARSED/INDEXING；全部向量成功后才原子更新 `active_version_id` 并置 READY。检索只连接活动 READY 版本。

**选择理由**：类似蓝绿发布，读路径始终有一个完整版本；失败补偿比跨数据库和模型 API 的长事务可靠。

**验证**：重复内容按 SHA-256 复用版本；真实检索查询包含 owner、kb、active version 和 READY 过滤。

## 6. API 请求不适合执行完整模型流水线

**现象**：上传后同步调用 Embedding 和图谱抽取会占用请求连接，模型 429/超时也无法可靠恢复。

**方案**：新增 `ingestion_tasks`。上传/同步事务只完成安全保存、解析、切分和任务入队；后台 Worker 自动执行 Embedding、版本切换和图谱抽取。管理员前端轮询 taskId。

**选择理由**：当前规模使用 PostgreSQL 任务表和进程内 Worker，减少 Celery/RabbitMQ 运维成本；任务规模增长后可保持表契约并替换执行器。

**验证**：自动流水线测试从上传开始，最终任务 SUCCEEDED、向量存在、活动版本切换且图谱候选生成。

**剩余风险**：PDF 文本提取和初始切分仍在上传请求中执行。超大文件应进一步拆为 `PARSE_DOCUMENT` 后台阶段或迁移到独立 Worker 进程。

## 7. 多 Worker 必须避免重复执行

**现象**：多实例同时轮询 PENDING 任务时可能重复调用收费模型。

**方案**：使用 PostgreSQL `SELECT ... FOR UPDATE SKIP LOCKED` 原子抢占。任务先提交 RUNNING，再执行外部调用。

**选择理由**：数据库已经是任务事实源，不需要为 MVP 新增分布式队列；SKIP LOCKED 能支持多个 Worker 横向竞争。

**验证**：真实 PostgreSQL 迁移和 integration CI 通过；SQLite 测试验证业务状态机。

## 8. Worker 崩溃会留下永久 RUNNING

**现象**：进程在任务设为 RUNNING 后退出，任务不会再次被 PENDING 查询选中。

**方案**：加入任务租约。`started_at` 超过 `BACKGROUND_TASK_STALE_SECONDS` 的 RUNNING 任务可被重新抢占，记录 `WORKER_LEASE_EXPIRED` 并增加重试计数。

**选择理由**：不依赖进程内存判断存活；服务重启后仍能从数据库恢复。

**剩余风险**：长任务需要周期性 heartbeat，避免正常执行时间超过租约后被重复抢占。当前 Chunk 规模较小，后续应增加 `heartbeat_at`。

## 9. 重试必须区分瞬时错误和永久错误

**现象**：认证错误、非法内容和模型 429 如果使用相同重试策略，会造成无意义调用或过早失败。

**方案**：429、连接失败、超时和结构化输出异常进入 RETRY_WAIT，使用指数退避且最多 3 次；确定性参数/内容错误直接 FAILED。管理员可显式重试 FAILED，手工重试清零 retry_count。

**选择理由**：有限重试能吸收短暂故障，同时控制成本和任务堆积。

## 10. 分阶段重试不能重复已完成工作

**现象**：Embedding 已成功并切换 READY 后，图谱抽取失败；重试如果再次索引，会重复付费，或者因 READY 不在可索引状态而失败。

**方案**：Worker 读取任务指定的 source_version。版本已 READY 时跳过 Embedding，只继续幂等图谱抽取。

**选择理由**：任务是阶段组合，但每个阶段需要可恢复、可重复检查的完成标志。

**验证**：真实简历首次向量成功、图谱失败后重试没有重复生成向量，最终任务成功。

## 11. SQLAlchemy rollback 后访问 ORM 对象触发 MissingGreenlet

**现象**：图谱阶段异常后执行 rollback，再读取 `task.id`，SQLAlchemy 尝试懒加载过期属性，在非 greenlet 上下文抛出 `MissingGreenlet`，Worker 本身崩溃。

**根因**：事务提交/回滚会使 ORM 状态过期；异常处理中不应依赖可能触发 IO 的对象属性。

**方案**：进入外部处理前保存纯字符串 `task_id`；rollback 后使用 `session.get()` 重新加载任务，再写失败状态。

**选择理由**：异常路径使用稳定标识符，不依赖 ORM identity state。

**验证**：真实简历图谱首次失败时任务正确进入 FAILED/RETRY_WAIT，Worker 不再崩溃。

## 12. DeepSeek 结构化输出与 Schema 不一致

**现象**：Prompt 要求 JSON，但 DeepSeek 可能返回 Markdown fenced JSON；JSON mode 下又使用 `type/from/to`，而代码期望 `entity_type/source/predicate/target`。

**方案**：模型客户端支持 `response_format={"type":"json_object"}`；解析器安全提取最外层 JSON；Pydantic 使用严格 `AliasChoices` 映射实际字段，同时保留长度、置信度和列表数量约束；Prompt 明确标准字段名。

**选择理由**：只靠 Prompt 不构成数据契约；只做字符串替换又缺少类型和范围验证。JSON mode + Pydantic 是透明且可测试的边界。

**验证**：裸 JSON、fenced JSON、标准字段和 DeepSeek 别名均有测试；真实简历生成 5 个实体和 4 条关系。

## 13. 知识图谱不能脱离原文证据

**现象**：模型抽取的实体和关系可能错误或来自文档内的恶意指令。

**方案**：图谱默认写为 CANDIDATE；每个实体/关系绑定 source_version、chunk、locator 和 quote_hash；管理员接受后才转 ACTIVE，并写不可变 revision。关系两端无法消歧时不创建关系。

**选择理由**：知识库原文是事实层，图谱是可重建派生层；不能让模型输出直接成为事实。

## 14. 网页同步存在 SSRF 风险

**现象**：管理员提交 URL 后，后端抓取功能可能被用来访问 localhost、私网服务或云元数据地址。

**方案**：只允许 HTTP/HTTPS；拦截 localhost 和非 global IP；限制响应大小；拒绝自动重定向并要求复核；隐藏底层网络异常。

**选择理由**：即便接口仅管理员可用，也不能假设输入永远可信；重定向可能绕过第一次 URL 校验。

**剩余风险**：域名解析后的私网 IP 和 DNS rebinding 还需在连接层增加解析校验。

## 15. Git 项目导入不能扫描任意服务器路径

**现象**：接受任意本地路径会让接口读取服务器敏感文件；仓库也可能包含 `.env`、依赖和二进制产物。

**方案**：仓库路径必须位于 `GIT_IMPORT_ROOT`；文本扩展名使用白名单；排除 `.git`、`.env*`、node_modules、虚拟环境和构建目录；限制单文件大小并跳过非 UTF-8 文件；Git/文件扫描移入工作线程。

**选择理由**：白名单比不断补充黑名单更可控；阻塞的文件系统和子进程调用不能占用异步事件循环。

## 16. SSE 不能只流式输出而不保存状态

**现象**：客户端断开或模型中途失败时，如果只写 HTTP 流，数据库无法判断回答是否完整，引用也无法审计。

**方案**：先创建 GENERATING 助手消息，独立保存 message_citation；正常结束置 COMPLETED，异常保存部分内容并置 FAILED；SSE 使用 metadata/citation/delta/done/error 稳定事件。

**选择理由**：传输状态和业务状态分离，便于恢复、统计和前端处理。

## 17. Docker 容器不能使用宿主机 localhost 连接依赖

**现象**：本地 `.env` 的数据库地址是 localhost，Compose 原样注入后，backend 容器会连接自己而不是 postgres 容器。

**方案**：Compose 显式覆盖容器内 `DATABASE_URL`、`REDIS_HOST` 和上传路径，使用服务名 `postgres`/`redis`；本地进程继续使用 localhost。

**选择理由**：同一份开发 `.env` 可服务宿主机工具，容器网络地址由部署层负责。

## 18. Integration CI 测试成功但清理步骤失败

**现象**：真实 pgvector/Redis 测试通过，GitHub Actions job 仍失败。

**根因**：`docker compose down` 解析完整 Compose 时发现仓库没有 `.env`，在清理阶段返回 1。

**方案**：integration job 启动前创建被忽略的空 `.env`；清理步骤使用 `if: always()`，始终删除容器和数据卷。

**选择理由**：测试成功不等于 job 成功；资源清理必须在成功和失败路径都可靠。

## 19. 可观测中间件顺序影响 requestId

**现象**：限流中间件位于 requestId 外层时，429 响应没有 requestId，无法关联日志。

**方案**：调整 Starlette 中间件注册顺序，使 RequestContext 位于最外层；限流直接返回统一 JSON 和 Retry-After。

**选择理由**：所有响应，包括中间件提前返回的响应，都应遵守错误和追踪契约。

## 20. 测试数据与真实效果必须区分

**现象**：Fake Retriever 的评测框架可以得到 100%，但这不代表真实模型效果；Windows 后台终端压测也没有稳定输出。

**方案**：文档明确标注 Fake 指标只验证框架；真实 pgvector/Redis 使用 Docker integration；真实检索评测通过 CLI 对固定数据集执行；无法可靠采集的压测不记录数字。

**选择理由**：只有可复现的同环境对照数据才能用于上线或简历结论。

## 23. 访客入口不能等同于用户注册

**现象**：产品需要让访客进入网站对话，但不希望访客注册账号或创建自己的知识库。

**根因**：公开注册会暗示系统是多用户知识库产品，并带来账号管理、密码找回和用户数据生命周期问题。

**方案**：新增 `POST /auth/access`。访客输入站点密钥后，后端创建独立 `USER` 身份并签发 JWT；前端只展示密钥入口。管理员使用右上角入口调用 `POST /auth/admin/login`，账号密码来自环境变量。

**选择理由**：访客会话仍由后端 Token 隔离，不依赖前端本地判断；每次密钥进入创建独立访客身份，避免不同访客互相读取会话。

**验证**：错误密钥返回 401；访客 A 无法读取访客 B 的会话；管理员登录后角色为 `ADMIN`。

## 24. 评论区必须进入后台而不是假界面

**现象**：对话窗口下方需要评论与岗位输入，并让管理员端操作简洁明晰。

**方案**：新增 `visitor_feedback` 表和 `/conversations/feedback` 接口。访客可提交岗位和评论，管理员端新增评论标签页查看最新反馈。

**选择理由**：评论是对个人成长有价值的数据，应持久化并纳入管理员端，而不是只在前端显示。

**验证**：自动化覆盖评论提交、管理员读取、会话删除后评论保留并将会话引用置空；默认测试更新为 `54 passed, 1 skipped`。

## 25. 本地文件入库一直停在 PENDING · 0%

**现象**：管理员端上传文件后，界面持续显示“入库任务 PENDING · 0%”，轮询接口正常返回但进度不前进。

**根因**：Docker 后端容器没有启用后台入库 Worker。上传/同步接口只负责保存来源、解析切分并创建 `ingestion_tasks`；Embedding、活动版本切换和图谱抽取必须由 Worker 消费任务。`BACKGROUND_WORKER_ENABLED` 未配置或容器未重启时，任务会一直保留在 `PENDING`。

**方案**：Compose 为 backend 显式设置 `BACKGROUND_WORKER_ENABLED=true`，本地 `.env` 同步保留该值；修改后重启 backend 容器。排障时进入容器执行配置检查，确认 `background_worker_enabled=True`，再查看 `ingestion_tasks` 是否从 `PENDING` 进入 `RUNNING/SUCCEEDED/FAILED`。

**选择理由**：保持 API 请求短事务，不在上传请求里直接调用模型；由后台 Worker 处理耗时、可重试和可恢复的模型流水线，是当前项目的核心入库设计。

**验证**：本地重启 backend 后，历史 3 个入库任务均被 Worker 消费并变为 `SUCCEEDED 100%`。

## 26. 本地 Git 同步提示 Git import root is not configured

**现象**：管理员端同步 Git 项目时报错 `Git import root is not configured`。

**根因**：后端 Git Connector 必须通过 `GIT_IMPORT_ROOT` 限制可读取目录；未配置时拒绝导入，避免接口读取服务器任意路径。

**方案**：Compose 将宿主机 `GIT_IMPORT_HOST_ROOT` 以只读方式挂载到容器内 `/app/git-imports`，并设置 `GIT_IMPORT_ROOT=/app/git-imports`。本地默认只挂载当前项目目录，因此同步本项目可填写 `/app/git-imports`；需要导入其他项目时，临时把 `GIT_IMPORT_HOST_ROOT` 指向该项目目录。

**选择理由**：只读挂载和容器内路径白名单同时生效；默认不挂载包含令牌、桌面资料或下载目录的上级目录，降低敏感文件暴露面。

**验证**：容器内配置检查显示 `git_import_root=/app/git-imports`；前端在未选择知识库或未填写路径时不再发送 `/knowledge-bases//sources:git` 这类空 ID 请求。

**后续补充**：`python:3.12-slim` 基础镜像默认不包含 `git` 命令，即使 `GIT_IMPORT_ROOT` 配置正确也会在读取 revision 时失败。后端 Dockerfile 安装最小运行时依赖 `git`，保证容器环境与 Git Connector 功能一致。

## 27. 大 Git 项目同步长时间显示 RUNNING

**现象**：同步 `/app/git-imports` 后，任务长期显示 `RUNNING`，进度停在 70% 之后；数据库中来源版本已经是 READY，Chunk 和向量已生成。

**根因**：入库 Worker 把“向量入库”和“DeepSeek 图谱候选抽取”串在同一个任务成功条件里。Git 项目会生成上百个 Chunk，图谱抽取需要逐个调用聊天模型，耗时高且可能因结构化 JSON 不合规触发 `GRAPH_EXTRACTION_INVALID`，导致用户误以为问答知识仍不可用。

**方案**：调整任务边界：Embedding 和活动版本切换完成即视为入库成功；图谱候选抽取作为增强步骤执行，受 `GRAPH_EXTRACTION_MAX_CHUNKS_PER_TASK` 限制，失败、跳过或部分完成只写入任务 warning，不再阻塞 `SUCCEEDED`。前端在任务成功但带 warning 时提示“问答知识已可用，图谱候选部分跳过或失败”。

**选择理由**：问答主链路依赖 pgvector RAG，不依赖图谱候选全部生成。把图谱抽取变成增强步骤，可以显著降低本地测试等待时间和模型成本，同时保留管理员后续审核知识图谱的能力。

**验证**：新增测试覆盖图谱结构化输出失败仍然完成向量入库，以及每个任务最多抽取指定数量 Chunk；相关测试 `9 passed`。

## 28. 源码噪声导致用户问答命中率低

**现象**：导入简历和整个项目目录后，访客询问“你做过哪些项目”，模型检索到 Python/FastAPI/异步编程等泛化技术说明，回答过度保守，无法给出清晰项目经历。

**根因**：早期 Git 导入直接拼接支持扩展名的源码和文档，源码、测试、八股资料、UI 文案、配置和真实项目经历混在同一个向量空间。RAG 主链路只做向量 Top-K，缺少面向“求职个人 Agent”的数据清洗、来源权重和问题扩展。

**方案**：新增入库清洗模块；PDF/文本清理页码、重复空白和重复行；Git 导入支持本地文件夹与公开 GitHub URL，但不再导入完整源码，而是生成项目资料卡、关键文档、配置摘要和代码结构摘要；检索阶段增加问题扩展、关键词重排和来源类型加权；Prompt 改为面向 HR/访客的高效回答，不展示无关上下文。

**选择理由**：对个人求职 Agent 来说，源码是证据来源，不是直接面向访客展示的知识。先把源码整理成“项目事实资产”，再入库检索，比让 LLM 在大量源码噪声里临场归纳更稳定、更省 Token，也更符合用户期待。

**验证**：后端全量测试 `56 passed, 1 skipped`；前端生产构建通过；本地数据库已清空旧导入内容，保留知识库、会话和评论，等待重新导入干净数据。

## 29. 访客端评论区位置与可见性调整

**现象**：评论区放在聊天窗口底部会挤占对话空间，且访客看不到其他人基于岗位留下的建议。

**方案**：评论区移动到左侧对话侧边栏下方，展示岗位和评论流；访客可提交并查看评论列表，管理员仍可在后台评论页查看完整反馈。

**选择理由**：评论是辅助反馈，不应打断主问答阅读；侧边栏适合承载对话列表和轻量社区反馈。

## 30. GitHub 仓库同步只返回 Unable to clone

**现象**：管理员同步 GitHub 仓库时接口返回 422 和笼统的 `Unable to clone GitHub repository`，无法判断是复制了 `/tree/...` 页面地址、仓库不存在、私有仓库未授权、网络失败还是超时。

**根因**：连接器只识别严格的 `https://github.com/...`，克隆异常又统一丢弃 Git 的失败类别；Git 在认证场景下还可能等待交互输入。

**方案**：新增 GitHub 地址规范化，接受 HTTPS、`.git`、`/tree/分支/...`、省略协议和 GitHub SSH 复制格式，并统一保存 HTTPS 仓库定位符；克隆设置 `GIT_TERMINAL_PROMPT=0`，将格式错误、仓库不存在/私有、网络失败和超时映射为不泄露底层 stderr 的中文错误。

**选择理由**：地址规范化减少小白用户复制网页路径导致的失败；错误分类应可操作，但不能把可能含凭据的完整 Git 输出返回前端。

**验证**：4 类常见地址规范化测试与 3 类无效地址测试通过；容器真实同步公开仓库的 `/tree/main/backend` 地址成功，创建 22 个 Chunk 和后台任务，无效单段 GitHub 地址返回明确 422。

## 31. SSE 已到达但页面最后一次性显示

**现象**：后端和 Nginx 已关闭缓冲，浏览器也收到多个 `delta`，但 Vue 页面可能等回答结束后才整段显示；纯文本展示无法清晰呈现项目列表和技术栈层次。

**根因**：新回复先以普通对象插入响应式数组，后续持续修改原对象字段，增量不一定逐次触发视图更新；同一网络读取中的多个 SSE 事件也会被 Vue 合并到一次渲染。

**方案**：SSE 解析器兼容 CRLF、跨分片和多行 `data`；每个文本增量进入显示队列，按浏览器动画帧替换响应式消息对象；等待期间显示生成光标并自动滚动。回答使用 `marked` 解析 Markdown，再由 `DOMPurify` 清洗后渲染。

**选择理由**：服务端流式传输、前端响应式更新和可读排版是三个独立环节，必须分别保证；Markdown 不能直接通过 `v-html` 信任模型输出。

**验证**：真实 DeepSeek 请求收到 81 个 `delta` 和 `done`；浏览器采样的回答长度按 `0 -> 3 -> 5 -> 23 -> ...` 连续增长，最终生成列表且不存在 `script` 节点。

## 32. Frontend 容器可访问但健康状态为 unhealthy

**现象**：网站通过反向代理返回 200，但 Compose 显示 frontend 为 `unhealthy`。

**根因**：Alpine 容器中的 `wget http://localhost/` 优先解析到 IPv6 `::1`，而该 Nginx 配置只监听 IPv4 80 端口，健康检查得到 connection refused。

**方案**：健康检查显式访问 `http://127.0.0.1/`。

**选择理由**：健康检查应验证实际监听地址，不能依赖不同镜像中的 localhost 地址族解析顺序。

## 33. 已导入内容不可见且无法单独维护

**现象**：管理员可以上传或同步，但看不到已导入文件、网页和 Git 项目；更新只能重复填写地址，删除只能清空整个知识库，无法批量导入或删除选定文件。

**方案**：增加批量文件端点和来源管理 API。来源列表聚合最新版本、Chunk 数、任务状态和更新时间；网页/Git 使用保存的 locator 重新同步；删除来源时显式清理版本、Chunk、任务、消息引用、图谱证据和存储文件，并只删除失去全部证据的图谱候选。前端按三类来源分栏，文件支持勾选批量删除。

**选择理由**：知识源是管理员日常维护的核心资产，不能只暴露“导入动作”。逐来源版本化和删除使更新范围明确，也避免为删一个错误文件而清空整个知识库。

**验证**：自动化覆盖 2 个有效文件加 1 个无效文件的部分成功、来源列表、文件删除与磁盘清理、网页二次同步生成新版本和来源删除。

## 34. RAG 在空知识库上生成不存在的项目经历

**现象**：已导入简历和两个 Git 仓库，但访客询问项目或技术栈时回答出现自动化考勤、电商推荐、Neo4j 问答平台、React、MongoDB 等资料中不存在的事实，且首段等待时间长。

**根因**：数据库同时存在两个已发布知识库，公开 Agent 固定选择较早的空知识库，模型实际收到空 Context；Prompt 又允许“合理归纳”，导致 DeepSeek 使用模型记忆补全。切换到有数据的知识库后，项目 2 的 52 个 Chunk 仍混入人物 Skill、志愿填报和压测文档，向量 Top-K 还被单一来源占满。常见问题每次调用远程 Query Embedding，增加约 9 秒冷延迟。

**方案**：发布一个知识库时事务内取消其他知识库；无检索结果时返回确定性拒答且不调用模型；Prompt 要求项目名和所有事实必须在 Context 中逐字有据。项目/技能/岗位问题改走无需远程 Query Embedding 的结构化快速检索，并按来源轮询选择简历和每个 Git 项目；上下文限制 6000 字符。Git 清洗器只选高信号项目文档，排除人物 Skill、Prompt、志愿资料和压测报告，清洗器版本变化可触发同一 Commit 重建版本。

**选择理由**：幻觉首先是错误知识库路由和无证据生成问题，不能靠提高 Top-K 或继续放宽 Prompt 修复；常见个人画像问题有明确来源结构，直接按来源取证比远程语义查询更快、更稳定。

**验证**：新增测试确保同一管理员仅一个知识库发布、空结果不调用聊天模型、项目问题不调用 Embedding 且前三条证据覆盖简历和两个 Git 来源。全量测试更新为 `68 passed, 1 skipped`。
