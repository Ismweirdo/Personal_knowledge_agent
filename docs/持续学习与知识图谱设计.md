# 持续学习与知识图谱设计

## 1. 设计原则

1. 原文优先：来源文件和不可变版本是事实依据。
2. 全链路溯源：实体、关系、摘要和回答必须能定位到来源版本与 Chunk。
3. 受控学习：模型只产生候选知识，规则决定自动接受或人工审核。
4. 可撤销重建：所有接受、合并、拒绝和失效操作记录修订事件。
5. 角色隔离：仅管理员可改变知识；访客仅拥有自己的会话。向量与图查询使用管理员知识 owner，消息查询使用访客 `user_id`。

## 2. 来源与版本

文件使用 SHA-256，网页优先使用 ETag/Last-Modified 并回退正文哈希，Git 项目使用 commit SHA 与单文件哈希。每次变化创建不可变 `source_version`。新版本完成解析和向量化后切换 `active_version_id`，问答检索先可用；图谱候选是可重建的派生增强层，失败或部分完成不阻塞来源版本发布，管理员可后续继续审核和补充。

项目导入默认允许 `.py`、`.java`、`.js`、`.ts`、`.vue`、`.md`、`.yml`、`.yaml`、`.sql` 等文本文件，排除 `.git`、`.env*`、密钥、依赖目录、构建产物、日志、数据库和大二进制文件。公开仓库不得包含用户上传的私人资料。

## 3. 图谱生成

每个 Chunk 经结构化 Prompt 提取候选实体和关系，输出通过 Pydantic Schema 校验。实体先做规范化、别名精确匹配，再结合类型、邻居和 Embedding 做消歧。无法可靠合并时创建新候选，不能强行覆盖。

关系结构为 `source_entity - predicate -> target_entity`，附带置信度、有效状态和一个或多个证据。只有存在有效证据的关系能参与回答。模型推断但无原文证据的内容标记为 `INFERRED`，默认不作为事实上下文。

## 4. 审核与冲突

- 高置信度、无冲突且来自可信来源的候选可自动接受。
- 涉及个人身份、时间、数量、结论变化或来源冲突的候选必须审核。
- 接受、拒绝、合并、拆分、失效和恢复均写入 `knowledge_revision`。
- 新版本不能物理覆盖旧事实；通过有效期和状态表达演进。

## 5. 学习闭环

上传资料、完成问答、确认候选、完成复习和修改项目都会形成 `learning_event`。掌握度只根据明确行为更新，不根据模型主观判断。复习任务使用可替换的间隔算法，初期可采用简化 SM-2。

Agent 可回答：最近学习内容、某技术出现在哪些项目、知识之间的关系、证据来源、待确认冲突和即将复习的主题。Agent 不允许自行修改源文件、删除证据或绕过审核策略。

## 6. 验收指标

- 实体抽取准确率、关系准确率和实体合并准确率。
- 图谱证据覆盖率与失效证据清理正确率。
- 增量同步跳过率、失败回滚成功率和版本切换耗时。
- RAG Recall@K、引用命中率、拒答正确率。
- 候选自动接受准确率、人工驳回率和冲突发现率。

模型结构化输出、实体消歧、证据幂等和审核实现中遇到的实际问题见 [开发文档](开发文档.md)。
