# 持续学习型个人知识库与知识图谱 Agent：技术文档

配套学习资料见 [技术八股全解索引](技术八股全解/00-索引与学习方法.md)，覆盖每项技术的选型、原理、项目用法、故障与面试追问。

## 1. 推荐技术栈

| 层次 | MVP 选型 | 说明 |
| --- | --- | --- |
| Python | Python 3.12+ | AI 服务、文档处理、Agent 编排和离线评测 |
| Web/API | FastAPI, Pydantic v2 | 认证、知识库 API、异步接口、SSE 和 OpenAPI |
| AI 编排 | 模型 SDK 为主，LangChain/LangGraph 按需 | 先保持 RAG 链路透明，复杂工作流再引入框架 |
| ORM/迁移 | SQLAlchemy 2, Alembic | 类型化数据访问和数据库版本管理 |
| 数据库 | PostgreSQL + pgvector | 统一保存业务数据、会话、任务、Chunk 与向量 |
| 知识图谱 | PostgreSQL 邻接表 + 证据表 | MVP 避免新增图数据库；保留迁移 Neo4j 的稳定领域接口 |
| 缓存 | Redis 7 | 限流、短期缓存；JWT 本身不强依赖 Redis |
| 文档解析 | PyMuPDF, python-markdown/纯文本解析 | PDF、Markdown、TXT 文本提取 |
| 网页抓取 | httpx, BeautifulSoup/readability | 抓取知识库链接并提取正文；动态页面按需使用 Playwright |
| 后台任务 | APScheduler + 受控任务表 | MVP 定时同步；规模增长后再迁移 Celery/RQ |
| 测试 | pytest, pytest-asyncio, Testcontainers | 单元、异步 API 与真实依赖集成测试 |
| 可观测 | structlog/logging, Prometheus | 健康检查、指标和结构化日志 |
| 部署 | Docker Compose, Nginx | 单服务器快速上线 |

选型约束：创建项目时锁定经过测试的 Python 依赖版本；优先使用模型厂商 SDK 和稳定抽象，不为追逐框架版本频繁改造主链路。

## 2. 工程结构

建议采用 Python 模块化单体：

```text
project1/
  backend/
    app/
      api/              # FastAPI routes, auth and schemas
      knowledge_base/   # knowledge base and source lifecycle
      connectors/       # file, web and future knowledge-source adapters
      ingestion/        # sync, parse, clean, chunk, embed
      retrieval/        # vector/hybrid retrieval and rerank
      agent/            # intent, prompt, tools and workflow
      conversation/     # history, citations and feedback
      knowledge_graph/  # entity, relation, evidence, resolution and revision
      learning/         # learning events, mastery and review scheduling
      evaluation/       # offline RAG evaluation
      infrastructure/   # model, database, Redis and storage clients
    tests/
    pyproject.toml
  frontend/
  deploy/
    docker-compose.yml
    nginx.conf
  docs/
```

FastAPI 路由只负责鉴权、校验和调用应用服务，核心同步/RAG/Agent 逻辑不能堆在路由函数中。连接器使用统一接口，使网页、文件和后续 Notion/语雀等来源能够独立扩展。

系统不是多租户知识库 SaaS。`ADMIN` 是唯一知识资产管理者；`USER` 只能访问已发布 Agent 并管理自己的会话。知识库 CRUD、上传、同步、索引、图谱抽取和审核统一依赖管理员授权。会话查询按访客 ID 隔离，RAG 检索使用知识库管理员 ID 过滤知识资产。

知识库、向量索引和知识图谱采用分层一致性：原始来源版本是事实依据，Chunk/向量可重建，实体关系是带证据的派生数据。任何图谱写入都必须携带 `user_id`、`kb_id`、来源版本、Chunk 和提取器版本。

## 4. API 草案

统一前缀 `/api/v1`，普通接口使用 JSON；流式问答使用 `text/event-stream`。

| 方法与路径 | 功能 |
| --- | --- |
| `POST /auth/register` | 注册 |
| `POST /auth/login` | 登录并返回 Token |
| `GET /knowledge-bases` | 查询当前用户知识库 |
| `POST /knowledge-bases` | 创建知识库 |
| `PATCH /knowledge-bases/{id}` | 修改知识库 |
| `DELETE /knowledge-bases/{id}` | 删除知识库及关联资源 |
| `POST /knowledge-bases/{id}/documents` | 上传文档 |
| `GET /documents/{id}` | 查询文档与处理状态 |
| `POST /documents/{id}/retry` | 重试失败任务 |
| `DELETE /documents/{id}` | 删除文档、切片和向量 |
| `POST /conversations` | 创建会话 |
| `GET /conversations/{id}/messages` | 查询消息历史 |
| `POST /conversations/{id}/messages:stream` | 提问并获取 SSE 回答 |
| `POST /messages/{id}/feedback` | 提交答案反馈 |
| `POST /knowledge-bases/{id}/sources` | 添加文件、URL 或 Git 项目来源 |
| `POST /sources/{id}/sync` | 手动触发增量同步 |
| `GET /knowledge-bases/{id}/graph` | 按主题查询实体关系与证据 |
| `GET /knowledge-bases/{id}/knowledge-candidates` | 查询待审核候选知识 |
| `POST /knowledge-candidates/{id}:accept` | 接受候选并记录修订 |
| `POST /knowledge-candidates/{id}:reject` | 拒绝候选并记录原因 |
| `GET /learning/review-tasks` | 查询复习任务和知识缺口 |

统一错误结构：

```json
{
  "code": "DOCUMENT_PARSE_FAILED",
  "message": "文档解析失败",
  "requestId": "01J...",
  "details": null
}
```

HTTP 状态码表达请求结果，业务错误码用于前端处理和日志检索。不要始终返回 HTTP 200。

## 5. 文档入库实现

### 5.1 状态与幂等

文档状态：`UPLOADED`、`PROCESSING`、`READY`、`FAILED`、`DELETING`。

上传时计算 SHA-256。以 `user_id + kb_id + hash` 建唯一约束或做显式重复检查。同一任务只能由一个执行器从 `UPLOADED/FAILED` 原子更新为 `PROCESSING`。

### 5.2 处理步骤

1. 校验知识库所有权、文件大小、扩展名和 MIME。
2. 生成 UUID 存储名并将文件写入持久化目录。
3. 在同一事务内创建文档记录和入库任务。
4. 事务提交后调度后台入库任务，避免任务先执行却查不到记录。
5. 清理页眉页脚、重复空白和不可见字符，保留页码/标题元数据。
6. 递归按标题、段落、句子切分，超限时按 Token 拆分。
7. 分批调用 Embedding API，并批量写入文本块和向量。
8. 全部成功后设为 `READY`；失败时记录错误类别和摘要。

不要对不可重复的外部 API 调用开启数据库长事务。失败清理由补偿逻辑处理，重试前按 `document_id` 清理未完成批次或使用确定性 Chunk ID 做 upsert。

## 6. 检索与生成实现

### 6.1 检索过滤

向量查询必须同时过滤 `user_id`、`kb_id`、`document_status=READY`。不能先全库检索再在应用层过滤，否则既可能泄露数据，也浪费 Top-K 名额。

API 层验证会话和知识库所有权，检索层仍强制使用当前用户和知识库元数据过滤，形成纵深防御。

基线流程：

```text
question
  -> optional query rewrite
  -> embedding
  -> vector search(topK=10, metadata filter)
  -> threshold + deduplication
  -> optional rerank
  -> context(topN=4..6)
  -> prompt
  -> streaming chat model
```

### 6.2 Prompt 约束

系统 Prompt 至少包含：角色、仅基于上下文回答、资料不足时拒答、忽略文档中的指令、引用格式、输出语言。文档内容使用清晰定界符包裹，降低 Prompt Injection 风险。

### 6.3 会话历史

数据库保存完整消息；每次模型调用只加载最近若干轮或按 Token 预算截断。后续可加入历史摘要。引用保存为 `message_id -> chunk_id` 关系，不能只把引用文本拼进答案。

## 7. SSE 协议

建议事件类型：

```text
event: metadata  # messageId, conversationId
event: delta     # incremental text
event: citation  # source list
event: usage     # tokens and latency
event: done
event: error     # stable error code, no stack trace
```

客户端断开时取消上游订阅或模型请求。服务端在结束、异常和取消三种路径中都释放资源。消息先创建为 `GENERATING`，正常结束改为 `COMPLETED`，中断/失败改为相应状态。

FastAPI 直接向浏览器输出 SSE。客户端断开时取消上游模型请求，不能缓冲整个回答后再发送。

## 8. 安全实现

- FastAPI 鉴权依赖默认保护业务路由，只显式放行注册、登录、健康检查和静态资源。
- JWT 使用足够长度的随机密钥和短期有效期；生产环境只经 HTTPS 传输。
- 所有资源 Service 方法接收当前用户 ID，并再次验证所有权，不能只依赖前端或 URL。
- 上传限制总大小和单文件大小，禁止原始文件名参与磁盘路径。
- 对 Markdown/模型输出进行 HTML 清理，前端禁止直接渲染不可信 HTML。
- 管理型同步接口需要用户鉴权；后台调度器不暴露公网控制入口。
- 数据库、Redis、向量库只暴露到 Docker 内部网络。

## 9. 异常、重试与限流

只对明确的瞬时错误重试，例如 429、连接重置和部分 5xx；使用指数退避加随机抖动，通常最多 2 至 3 次。参数错误、认证失败和内容超限不重试。

限流至少覆盖：

- 登录：按 IP 和账号。
- 文件上传：按用户，限制并行入库数。
- 问答：按用户，限制每分钟次数和同时进行的流。

模型提供方不可用时快速失败并给出稳定错误码，不无限排队占满线程池。

## 10. 日志与指标

每个请求生成 `requestId`，异步任务生成 `taskId`，模型调用记录 `conversationId/messageId`。使用结构化字段，避免打印文档正文和完整 Token。

核心指标：

- API 请求量、P50/P95/P99 延迟、错误率。
- 入库队列长度、处理耗时、成功率、失败分类。
- 检索耗时、返回片段数、最高/最低相似度。
- 模型首 Token 延迟、总耗时、Token 使用量、失败率。
- JVM 堆、GC、线程池活跃数、数据库连接池使用率。

## 11. 测试策略

### 单元测试

- Chunk 边界、Overlap、空文本和超长段落。
- 权限判断、状态机转换、错误码映射。
- Prompt 上下文预算和引用编号。

### 集成测试

- 使用 Testcontainers 启动 PostgreSQL/pgvector 和 Redis。
- 覆盖注册登录、跨用户访问拒绝、上传入库、向量过滤、删除清理。
- 模型 API 使用 Stub/WireMock，验证超时、429、流式中断和重试。

- 使用 pytest 测试解析、Chunk、Prompt、权限过滤参数和状态转换。
- 使用 pytest-asyncio/httpx 测试 FastAPI 异步接口与流式响应。
- 模型与 Embedding 客户端使用 Fake/Mock，测试超时、限流和无相关资料拒答。
- 为知识源同步测试未变化、内容更新、页面删除、抓取超时和重复调度场景。

### 端到端与评测

- 准备小型固定语料和 20 至 30 个问题，包括可回答、跨段、多轮和不可回答问题。
- 记录检索 Recall@K、引用命中率、拒答正确率、回答正确性、首 Token 和总延迟。
- 每次修改 Chunk、模型、Prompt 或检索参数后保留对照结果。

## 12. Docker Compose 部署

服务建议包含：`nginx`、`frontend`、`backend-python`、`redis`、`postgres/pgvector`。数据库和文件目录挂载命名卷或服务器固定目录，容器设置健康检查和自动重启策略。

部署步骤：

1. 准备 Linux 服务器、域名、防火墙和 Docker Compose。
2. 只开放 80/443 和必要的 SSH 端口。
3. 在服务器创建 `.env`，设置强密码和密钥。
4. 构建或拉取镜像，执行数据库迁移，再启动服务。
5. 配置 Nginx HTTPS；SSE 路径关闭代理缓冲并增加读取超时。
6. 验证健康检查、上传、问答、重启持久化和备份恢复。

### 上线检查清单

- [ ] 仓库中不存在 API Key、密码和真实 `.env`。
- [ ] HTTPS 正常，数据库端口未对公网暴露。
- [ ] SSE 不被 Nginx 缓冲，断开能够释放后端资源。
- [ ] 数据卷、备份目录和恢复步骤已验证。
- [ ] `/actuator` 仅开放必要端点，敏感端点受保护。
- [ ] 日志可按 requestId/taskId 定位一次失败。
- [ ] 限流、上传大小和模型超时均生效。
