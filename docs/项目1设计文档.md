# 持续学习型个人知识库与知识图谱 Agent：项目设计文档

## 1. 项目定位

### 当前实现基线（2026-07-11）

| 已实现并验证 | 后续优化，不能作为当前简历成果 |
| --- | --- |
| FastAPI 管理员/访客双端、JWT 与角色隔离 | 向量+关键词混合检索 |
| PDF/Markdown/TXT、网页和本地 Git 项目接入 | Reranker 与 Query Rewrite |
| ETag/Last-Modified、内容哈希和 Git revision 增量判断 | 通用工具调用与复杂 Agent 工作流 |
| PostgreSQL 任务表、SKIP LOCKED 抢占、租约恢复和指数退避 | OCR、Office 和动态网页 Playwright 抓取 |
| GitHub Models 1536 维 Embedding、pgvector 余弦检索 | 独立 Celery/RQ Worker 或专用向量数据库 |
| DeepSeek SSE 回答、引用持久化、会话隔离和拒答 Prompt | 正式服务器 HTTPS 与生产负载指标 |
| 证据化实体/关系抽取、候选审核、修订记录和复习任务 | 复杂图分析与 Neo4j |
| 52 项默认测试、真实 pgvector/Redis 验证、CI、Docker Compose | 大规模真实评测后的效果优化百分比 |

### 1.1 项目目标

构建一个代表“我”的可部署、可持续成长的个人 Agent。管理员本人接入简历、文档、网页、项目仓库和学习笔记；访客只能与该 Agent 对话，不能利用系统创建自己的知识库或 Agent。

系统不把模型输出直接视为事实。知识库是原文事实层，知识图谱是带证据、置信度和版本的派生层；Agent 提取的新知识必须经过去重、冲突检测和自动/人工审核策略后才能生效，并且可撤销、可重建。

项目既要解决真实的信息检索问题，也要集中体现 Python Agent 应用能力：知识源同步、文档处理、RAG、工具调用、工作流编排、流式响应、效果评测、测试、容器化和线上排障。Java 后端能力由项目 2 独立体现，避免项目 1 为展示技术栈而增加无必要的跨语言调用。

### 1.2 目标用户

- 管理员：知识主体本人，负责导入、审核、发布和持续更新个人知识。
- 普通用户：招聘者、同学或访客，通过问答了解管理员的经历、项目和知识观点。

### 1.3 成功标准

- 管理员可增量导入资料并审核候选知识；普通用户注册后只能开始对话。
- 回答基于知识库内容并展示可追溯来源；缺少依据时明确拒答。
- 每个图谱实体和关系可追溯到文档版本、Chunk、页码或代码位置。
- 文件、网页和 Git 项目变化后只处理差异，失败时继续提供上一可用版本。
- 用户可以查看最近学到的知识、待确认知识和需要复习的主题。
- 文档处理失败可定位原因并重试，不影响其他文档。
- 服务可通过 Docker Compose 部署到 Linux 服务器。
- 有固定评测集，关键效果与性能能够量化说明。

## 2. 范围

### 2.1 MVP 功能

1. 用户注册、登录和 JWT 鉴权。
2. 仅管理员可创建、查询、修改、发布和删除知识库。
3. 仅管理员可上传 PDF、Markdown、TXT 或同步网页/Git 项目。
4. 知识源手动同步和定时增量同步；源内容变化后更新对应切片与向量。
5. 基于指定知识库的检索增强问答和 AI 客服式多轮对话。
6. SSE 流式输出、会话历史、来源链接和引用片段。
7. 管理知识资产与访客会话隔离；访客之间的会话数据隔离。
8. `ADMIN/USER` 角色授权，所有知识写接口强制管理员权限。
8. 基本日志、异常处理、限流、健康检查。
9. Docker Compose 部署。
10. 从 Chunk 中提取实体、别名和关系，保留证据、置信度与提取模型版本。
11. 学习事件记录、候选知识审核、冲突处理、撤销与重建。
12. 基于图谱和学习事件进行关联查询、知识缺口提示与基础复习计划。
13. 本地目录/Git 项目按提交或文件哈希增量导入，并应用敏感文件排除规则。

### 2.2 MVP 不做

- 多智能体协作和无人监督自主规划。
- Office/图片 OCR 等复杂格式解析。
- 企业级组织、角色和计费系统。
- 模型训练或微调。
- 复杂 3D 图谱可视化和通用图分析平台。
- MVP 阶段独立部署 Neo4j；先使用 PostgreSQL 关系表，达到明确查询瓶颈后再同步到图数据库。

这些内容不能阻塞主链路上线。

## 3. 核心用户流程

### 3.1 文档入库

用户登录 -> 创建知识库 -> 上传文件或添加知识源链接 -> 创建同步任务 -> 抓取/解析内容 -> 计算内容哈希与变更 -> 清洗与切分 -> 生成向量 -> 替换发生变化的切片 -> 知识源状态变为可用。

### 3.2 知识问答

用户选择知识库并提问 -> 鉴权与限流 -> 意图识别/问题改写 -> 检索候选片段 -> 重排 -> 判断是否需要工具或人工转接 -> 构造上下文与 Prompt -> 调用模型 -> SSE 输出 -> 保存消息与引用 -> 用户反馈。

### 3.3 持续学习

知识源产生变化 -> 创建不可变来源版本 -> 解析并生成 Chunk -> 提取候选实体/关系 -> 实体消歧与证据绑定 -> 检测重复和冲突 -> 按置信度自动接受或进入待审核队列 -> 原子切换有效版本 -> 写入学习事件 -> 更新掌握度和复习计划。

## 4. 系统架构

```text
Browser / Web UI
        |
     Nginx
        |
Python Agent Application (FastAPI)
  |-- Auth / User / Knowledge Base
  |-- Source Connector / Sync Scheduler
  |-- Document Parse / Chunk
  |-- Embedding / Retrieval / Rerank
  |-- Prompt / Agent Workflow / Tools
  |-- Entity / Relation Extraction / Resolution
  |-- Learning Event / Review / Revision
  |-- Conversation / Citation / Feedback
  |-- Evaluation / Observability
        |
  +-----+-------------+---------------+
  |                   |               |
PostgreSQL         Redis        pgvector
metadata/session   cache/limit   chunks/embedding
  |                                   |
Object/File Storage              LLM/Embedding API
```

项目采用 Python/FastAPI 模块化单体。对当前个人知识库规模，单服务足以承载业务 API、同步任务和 Agent 工作流，能够显著减少开发、部署和排障成本。模块之间保持清晰边界，未来只有当抓取任务或推理负载形成独立扩缩容需求时才拆服务。

删除知识库时需要同步清理文档、文本块、向量和会话引用。MVP 可使用受控级联删除；生产化时改为异步清理并记录审计事件。

知识图谱记录不能脱离证据独立存在。来源版本失效时，对应证据失效；没有其他有效证据的实体或关系转为 `STALE`，而不是直接物理删除，以支持审计和回滚。

## 7. RAG 策略

### 7.1 基线方案

- 按标题/段落优先切分，再按 Token 上限拆分。
- 初始 Chunk 建议 500 至 800 Token，Overlap 80 至 120 Token，最终参数由评测决定。
- 问题生成向量后，在指定用户和知识库范围内检索 Top 10。
- 根据相似度阈值过滤，选择 Top 4 至 6 进入上下文。
- Prompt 明确要求只依据资料回答、资料不足则拒答、输出引用编号。

当前代码实现为 pgvector 余弦距离 Top 6 检索、活动 READY 版本过滤、来源引用持久化和资料不足拒答 Prompt。相似度阈值、关键词融合、去重、Reranker 和 Query Rewrite 尚未接入。

### 7.2 优化顺序

1. 建立评测集和基线。
2. 调整 Chunk、Overlap、Top-K 和阈值。
3. 加入关键词与向量混合检索。
4. 加入 Reranker。
5. 针对多轮对话增加 Query Rewrite。

没有评测数据时不引入复杂链路。

## 8. 非功能需求

### 安全

- 密码使用 BCrypt/Argon2 哈希；密钥只来自环境变量。
- 所有知识库、文档、会话查询都强制带 `user_id` 权限条件。
- 校验文件扩展名、MIME、大小和路径；生成服务端存储名。
- 日志不记录密码、Token、完整 Prompt 或敏感文档内容。
- 对登录、上传和问答接口分别限流。

### 可靠性

- 文档入库具备状态机：`UPLOADED -> PROCESSING -> READY/FAILED`。
- 使用文档哈希和任务状态保证重复提交幂等。
- 外部模型调用设置连接/读取超时、有限重试和失败记录。
- 定时同步使用分布式锁或任务唯一约束，防止同一知识源被重复处理。
- 链接同步优先比较 ETag/Last-Modified，无法获得时计算规范化正文哈希；内容未变化则跳过 Embedding。
- 更新时先构建新版本切片，全部成功后原子切换为当前版本；抓取或向量化失败时继续提供上一可用版本。
- 数据卷持久化；数据库定期备份。

### 性能

- 问答使用 SSE，尽快返回首 Token。
- 文档入库不占用 Web 请求线程；MVP 可使用受控线程池，后续可迁移到消息队列。
- 大文本分批 Embedding 和写入，避免内存峰值。
- 记录检索耗时、首 Token 延迟、总耗时和 Token 用量。

## 9. 验收测试

| 场景 | 验收条件 |
| --- | --- |
| 正常入库 | 支持格式的文档最终变为 READY，切片可检索 |
| 失败入库 | 错误格式/解析失败时状态为 FAILED 且有可读原因 |
| 权限隔离 | 用户 A 无法查询、下载或问答用户 B 的数据 |
| 有依据问答 | 回答包含正确结论和至少一个有效来源 |
| 无依据问答 | 不编造答案，明确说明资料不足 |
| 多轮问答 | 后续问题在有限历史窗口内保持语义连贯 |
| 流式异常 | 模型失败时 SSE 返回明确错误事件并保存失败状态 |
| 重启恢复 | 容器重启后数据库、文件和向量数据仍存在 |
| 图谱可追溯 | 实体和关系能定位到有效来源版本及原文片段 |
| 增量更新 | 未变化内容不重复向量化；变化内容成功后原子替换 |
| 冲突处理 | 新旧来源冲突时不静默覆盖，生成待审核项 |
| 撤销学习 | 错误候选可拒绝，已接受知识可撤销并留下修订记录 |
| 项目导入安全 | 默认排除 `.git`、`.env`、依赖、构建产物和二进制文件 |

## 11. 项目面试讲解重点

- 为什么项目 1 选择 Python 模块化单体，并用项目 2 单独证明 Java 后端能力。
- 如何检测网页/知识库链接变化并做到增量更新、删除旧向量和失败回滚。
- 文档切分参数如何通过评测确定。
- 如何保证用户数据在向量检索阶段仍然隔离。
- 模型超时、限流、流式中断和入库失败如何处理。
- 如何降低幻觉并让答案可追溯。
- 如果文档量和并发增加 100 倍，系统如何演进。

所有性能和效果提升均需保留测试环境、样本量、对照参数和原始结果。
