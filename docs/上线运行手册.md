# 上线运行手册

## 发布前

- 使用随机强 `JWT_SECRET`，配置数据库、Redis、DeepSeek 与 Embedding 密钥。
- 设置 `APP_ENV=production` 和 `RATE_LIMIT_ENABLED=true`。
- 运行 Alembic 迁移，并配置 `ADMIN_USERNAME`、`ADMIN_PASSWORD` 和 `VISITOR_ACCESS_KEY`。
- 确认 Nginx 将 `/api/` 代理到后端，SSE 路径关闭代理缓冲。
- 只开放 80/443；PostgreSQL、Redis 和 `/metrics` 不对公网开放。
- 确认私人上传目录、`.env`、备份目录不在 Git 仓库中。

## 推荐服务器

预期访问量很小时，`2 vCPU / 2 GiB / 40 GiB` 可以作为首个 beta 生产环境。由于本项目同时运行后端、前端、PostgreSQL/pgvector、Redis 和 Nginx，`2 GiB` 内存偏紧，建议创建 `2-4 GiB` swap，并持续观察内存、磁盘和数据库连接数。如果预算允许，`2 vCPU / 4 GiB` 是更稳妥的长期选择。

系统建议 Ubuntu 24.04 LTS。公网安全组只开放 SSH、HTTP 和 HTTPS；SSH 使用密钥登录，关闭密码登录。模型推理和 Embedding 走外部 API，不在服务器本地部署大模型。

## 首次部署

1. 安装 Docker Engine、Docker Compose Plugin、Git 和 Nginx/Certbot。
2. 拉取仓库代码，复制 `.env.example` 为 `.env`，只在服务器本地填写真实密钥。
3. 将 `APP_ENV=production`、`RATE_LIMIT_ENABLED=true`，并替换 `JWT_SECRET`、`POSTGRES_PASSWORD`、`REDIS_PASSWORD`、`LLM_API_KEY`、`EMBEDDING_API_KEY`、`ADMIN_USERNAME`、`ADMIN_PASSWORD` 和 `VISITOR_ACCESS_KEY`。
4. 执行 `docker compose -f deploy/docker-compose.yml up -d --build`。
5. 使用 `docker compose -f deploy/docker-compose.yml ps` 确认容器健康。
6. 配置域名解析到服务器公网 IP，用 Nginx 或云厂商证书服务启用 HTTPS。
7. 访问 `/health` 和 `/health/ready`，完成管理员登录、知识导入、访客问答、评论提交和重启持久化验证。

国内服务器访问 GitHub clone 链路可能较慢。系统会先尝试浅克隆，失败后自动改用 GitHub 仓库压缩包下载并走同一套清洗规则；若公开仓库仍超时，可先在服务器 `/opt/laylight/git-imports` 下手动 clone，再在管理员端填写容器路径 `/app/git-imports/仓库目录名`。

## 验证

1. `/health` 返回 200，`/health/ready` 的数据库和 Redis 均为 up。
2. 访客通过站点密钥进入，普通 USER 访问知识写接口返回 403。
3. 上传、索引、发布、访客 SSE 问答、引用链路和 HR 评论提交可用。
4. 管理员可查看知识库、候选审核、复习任务和访客评论。
5. 登录、上传和问答超过限额后返回 429 与 `Retry-After`。
6. 容器重启后数据库、上传文件、索引和评论仍存在。

## 备份恢复

每日执行 `deploy/backup-postgres.sh`，默认保留 14 天。上传文件卷需要使用服务器快照或对象存储版本控制另行备份。每月至少在隔离环境执行一次 `restore-postgres.sh`，并验证管理员登录、知识库数量、Chunk 数、问答引用和 `visitor_feedback` 评论记录。

## 故障处理

- 按 requestId 查询结构化请求日志，不记录 Token、Prompt 或文档正文。
- 就绪检查失败时先检查数据库连接、Redis 和迁移版本。
- 模型服务失败时保留旧知识版本，禁止删除现有活动向量后重试。
- 回滚应用镜像时确认其支持当前数据库迁移版本；数据库回滚必须先备份。

日常后台操作见 [管理员使用说明](管理员使用说明.md)。已知故障模式和历史解决过程见 [开发文档](开发文档.md)。生产出现新故障时，恢复服务后必须补充现象、根因、处置、验证和预防措施。
